### 方案 2: 优化 Agent 提示词（预期改善 20-30%）

**问题**: 模型每次工具调用后需要 5-27 秒思考

**优化方向**:
1. 减少不必要的系统提示词长度
2. 明确指示模型快速决策
3. 限制工具调用的最大次数
4. 添加"快速模式"提示

**实施步骤**:
```markdown
# 在 agent 系统提示中添加:

## 响应速度要求
- 简单问题直接回答，不要调用工具
- 复杂任务最多调用 2 次工具
- 每次决策控制在 3 秒内
- 优先使用已有知识，减少外部查询
```

**预期效果**: 减少模型犹豫时间，加快决策速度

---

### 方案 3: 并行工具调用（预期改善 30-40%）

**问题**: 当前工具调用是串行的
```
工具 1 (26.8 秒思考 + 18ms 执行)
  ↓
工具 2 (6.5 秒思考 + 17ms 执行)
  ↓
工具 3 (13.3 秒思考 + 12ms 执行)
```

**优化**: 对于独立任务，可以并行执行
```
工具 1 + 工具 2 + 工具 3 (同时执行)
  ↓
合并结果 (一次性)
```

**实施**: 需要修改 Agent 逻辑，支持并发工具调用

**预期效果**: 多次工具调用时间从累加变为取最大值

---

### 方案 4: 解决钉钉 webhook 推送延迟（预期改善 30%）

**问题**: 钉钉推送延迟 24 秒

**可能原因**:
1. 钉钉服务器到 Gateway 的网络延迟
2. webhook 回调配置问题
3. Gateway 接收队列堵塞

**排查步骤**:
```bash
# 1. 检查钉钉开放平台 webhook 配置
# 2. 测试网络延迟
curl -w "@curl-format.txt" -o /dev/null -s https://oapi.dingtalk.com/robot/send

# 3. 检查 Gateway 日志是否有队列积压
tail -100 /tmp/openclaw/openclaw-*.log | grep -E "(queue|dispatch|webhook)"

# 4. 对比飞书和钉钉的延迟差异
```

**解决方案**:
- 如果是网络问题：考虑使用更近的服务器
- 如果是配置问题：检查 webhook URL 是否正确
- 如果是队列问题：优化 Gateway 并发处理

---

### 方案 5: 会话上下文压缩（预期改善 10-20%）

**问题**: 长上下文增加模型推理时间

**优化**:
1. 启用更积极的上下文压缩
2. 定期清理旧消息
3. 使用摘要代替完整历史

**当前配置**:
```json
"compaction": {
  "mode": "safeguard"
}
```

**建议修改**:
```json
"compaction": {
  "mode": "aggressive",
  "maxMessages": 50,
  "compressAfter": 20
}
```
